# 从 Harness Engineering 到视频 AI 的质量飞轮

最近读了 OpenAI 的 [Harness Engineering](https://openai.com/index/harness-engineering/)，讲的是他们如何用 Codex agents 在五个月内从零构建一个百万行代码的产品，全程人类没有手写一行代码。

文章有很多洞察，但对我冲击最大的不是"agents 能写代码"这件事，而是他们面对的核心挑战和我们做视频 AI 产品时遇到的几乎一样：**当 AI 是主要的产出者时，人类的角色到底是什么？如何确保迭代方向是正确的？**

这篇笔记记录我从这篇文章出发，逐步推导出一套适用于我们视频 AI 产品的质量反馈系统的过程。

## 我们的痛点

我们的视频 AI 产品的核心 pipeline 是：

```
User Intent → Virtual Script (结构化 Shot Plan)
→ Keyframe 生成 (Nano Banana) → Transition Video (Motion Prompt) → 最终视频
```

Virtual Script 是"导演"，规划每个 shot；Nano Banana 生成 keyframe；相邻 keyframe 之间用 motion prompt 生成转场视频，最后拼接。

听起来清晰，实际做起来有三个致命问题：

**Prompt 蝴蝶效应。** 改了一个 keyframe 的 style prompt，下游所有 transition 可能跟着崩。更痛苦的是，你根本不知道这次改动整体是变好了还是变差了。

**人类反馈无法闭环。** 团队在 Slack 里说"这个转场太跳了"、"色调不对"，然后呢？下次迭代同样的问题再次出现。反馈没有被系统消费。

**头痛医头。** 改一个 prompt 修了一个问题，另一个问题又冒出来。缺乏系统性的改进方向。

## 从文章中提取的关键原理

Harness Engineering 的核心理念是 **"Humans steer. Agents execute."**，人类不再写代码，而是设计环境、明确意图、构建反馈循环。

其中有几条对我们直接适用：

**1. 不在仓库里的知识等于不存在。** 文章举了一个例子：团队在 Slack 上达成了一个架构共识，但如果这个信息没有进入代码仓库，对 agent 来说就跟不存在一样。我们的质量反馈也是如此——停留在聊天记录里的反馈，pipeline 读不到，等于不存在。

**2. 品味捕获一次，持续执行。** OpenAI 的做法是把人类的品味编码成 lint 规则和结构测试，然后自动执行到每一行代码。我们需要类似的机制：把人类对视频质量的判断编码下来，让系统能持续地检查。

**3. 修环境不修 agent。** 当 agent 表现不好时，正确的做法不是"再努力一点"，而是问"缺少什么能力？怎么让它对 agent 可读且可执行？"

## 推导过程中的几个关键转折

### 转折 1：LLM-as-judge 行不通，但 LLM-as-checker 可以

我们尝试过用 LLM 给视频打分（1-5 分），结果不可靠。一度以为自动化质量评估这条路走不通。

但在推导过程中发现一个关键区分：

```
LLM 做不好的事：  "这个视频质量打几分？"（主观、模糊、开放式）
LLM 能做好的事：  "之前人类说运动不连贯，现在还是吗？"（具体、二元、针对性）
```

LLM 不能当评委，但可以当检查员。给它一个人类定义好的具体问题，让它做 Yes/No 判断，这个任务足够窄，LLM 可以胜任。

这打开了整个系统设计的空间。

### 转折 2：不需要评分，不需要 checklist，用开放式评论

最初想设计一套 binary checklist（"运动是否连贯？Yes/No"，"是否符合物理规律？Yes/No"），但想了一下发现不对——这需要我们预先知道所有可能的问题类型，而且会限制 reviewer 的视角。

更好的方案：**让人类写开放式评论**，然后让 LLM 从大量评论中提取共性。

这样起步成本为零，不需要设计问题集合。reviewer 想到什么写什么，LLM 会从 200 条散落的评论中自动浮现出 "快速运动场景中人物姿态跳变出现了 23 次" 这样的通用问题。

### 转折 3：不需要元素级 Before/After 对比

一开始想做 Before/After 对比：改了 prompt 后，对同一组输入跑新旧两版，逐个 keyframe/transition 比较。但马上发现一个问题：LLM 产出是随机的。同一个输入跑两次，第一次可能生成 10 个 shot，第二次变成 8 个，keyframe 完全不同，根本无法一一对应。

想了一下，其实不需要。我们比较的不是"这个 keyframe 跟上次那个 keyframe 比如何"，而是"在同一批输入上，这类通用问题的出现率是下降了还是上升了"。这是统计层面的趋势比较，不需要元素级对应。

而且通用问题（"是否存在姿态跳变？"）天然是 shot-count agnostic 的——不管生成了几个 shot，都能回答。

### 转折 4：全链路可溯源，可以从问题反推到代码

我们的系统有一个天然优势：每次运行都会留下完整的计算图（所有中间产出）和 commit ID（代码版本）。这意味着当发现一个问题时，LLM 可以结合问题描述、计算图中的中间数据、和对应版本的代码，反向追溯根因。

甚至可以更进一步：让 LLM 直接提出修复方案，跑 Golden Cases 验证无 regression 后自动合并。这就是文章中 "Increasing levels of autonomy" 的落地形态。

## 最终系统设计

四个组件，两条工作流：

```
工作流 A: 日常反馈积累              工作流 B: 变更回归检测

线上 project 产出                  系统变更 (prompt/model/参数)
       |                                  |
       v                                  v
  Review 页面                        Golden Cases
  (可视化计算图,                     (30-50 个标准输入)
   开放式评论)                             |
       |                                  v
       v                           LLM Targeted Check
  反馈持久化                        (拿通用问题逐项检查)
       |                                  |
       v                                  v
  LLM 提炼通用问题 ───────────>   Regression Report
       |                           (问题出现率变化)
       v                                  |
  通用问题库                              v
  (持续增长)                    Root Cause Analysis
                                (计算图 + commit → 定位根因)
                                          |
                                          v
                                 Auto-Evolution
                                (自动修复 → 验证 → 合并)
```

**组件 1: Review 页面** — 每个线上 project 一个。可视化 Virtual Script → Keyframe → Transition 的计算图，人类在每个步骤留开放式评论。反馈锚定在核心步骤上，这三步是架构不变量，不会因功能迭代失效。

**组件 2: Golden Cases** — 30-50 个精选标准输入作为 benchmark。不需要完美复刻，统计层面的趋势比较即可。

**组件 3: LLM Targeted Check** — 从评论中提炼的通用问题，对 Golden Cases 新产出逐项做 Yes/No 检查。聚合为 Regression Report，给出量化的 confidence 信号。

**组件 4: Root Cause Analysis + Auto-Evolution** — 利用计算图 + commit ID 的全链路可溯源性，LLM 分析代码定位根因，提出修复甚至自动演进。

## 核心设计原则

几条在推导过程中逐步确立的原则：

**LLM 做检查员，不做评委。** LLM 不做通用质量评价，只做针对性验证。这是整个系统成立的关键区分。

**趋势正向 > 完美复刻。** 不追求每个 case 都变好。Baseline 上通用问题出现率下降就可以发布，个别退步可接受，记录为新问题进入下轮循环。

**反馈产生复利。** 每条人类评论被持久化，LLM 提炼出通用问题后，在未来每一次变更中自动检查。人类的品味判断被捕获一次，持续执行。

**全链路可溯源。** 每次运行留下计算图 + commit ID，问题可以从产出反向追溯到代码。这为自动化修复提供了基础。

## 几点反思

回头看，文章对我最大的启发不是具体的技术方案，而是一个思维方式的转变：

**不要调 prompt，调环境。**

当产出质量不好时，本能反应是去改 prompt。但正确的做法是退一步问：人类的反馈有没有被系统读到？质量标准有没有被编码下来？迭代方向有没有量化的信号？

如果这些基础设施不在，再怎么调 prompt 都是在四处救火。

文章里 OpenAI 团队也经历过类似的阶段——早期进展比预期慢，不是因为模型能力不够，而是"环境不够明确"。工程师的核心工作从"写代码/调 prompt"变成了"设计约束和反馈循环"。

这大概就是 Harness Engineering 的真正含义：不是驾驭 AI 去做事，而是构建让 AI 能可靠做事的脚手架。
